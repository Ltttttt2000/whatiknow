## RESTful API
**RESTful API定义**
```
Representational State Transfer 的缩写，翻译：“具象状态传输”。一般解释为“表现层状态转换”。
符合REST原则的API设计，使用HTTP协议定义了一组统一的接口，使用HTTP方法（GET、POST、PUT、DELETE）用于访问和操作资源。使用URL来唯一标识资源。
REST 是设计风格而不是标准。是指客户端和服务器的交互形式。
```
**REST的特点**
```
(1) 具象的。一般指表现层，要表现的对象就是资源。比如，客户端访问服务器，获取的数据就是资源。比如文字、图片、音视频等。
(2) 表现：资源的表现形式。txt 格式、html 格式、json 格式、jpg 格式等。浏览器通过 URL 确定资源的位置，但是需要在 HTTP 请求头中，用 Accept 和 Content-Type 字段指定，这两个字段是对资源表现的描述。
(3) 状态转换：客户端和服务器交互的过程。在这个过程中，一定会有数据和状态的转化，这种转化叫做状态转换。其中，GET 表示获取资源，POST 表示新建资源，PUT 表示更新资源，DELETE 表示删除资源。HTTP 协议中最常用的就是这四种操作方式。 
```
**RESTful API的特点是什么？**
```
基于HTTP协议：使用HTTP方法对资源进行操作。
资源标识：使用URL唯一标识资源。每个 URL 代表一种资源。
统一接口：使用统一的接口对资源进行访问和操作。
无状态性：每个请求都是独立的，服务器不保存请求的状态信息。
```
**RESTful API的设计原则是什么？**
```
资源和标识：每个资源都有唯一的URL标识。
使用HTTP方法：使用HTTP方法对资源进行操作，如GET（获取资源）、POST（创建资源）、PUT（更新资源）、DELETE（删除资源）。客户端通过四个 http 动词，对服务器资源进行操作，实现表现层状态转换。
状态转移：通过资源的状态转移来实现业务逻辑。客户端和服务器之间，传递这种资源的某种表现层。
无状态性：每个请求都是独立的，服务器不保存请求的状态信息。
```
**RESTful API的优缺点是什么？**
```
优点：
    - 易于理解和使用：使用HTTP协议和URL来定义接口。
    - 可扩展性：可以通过增加资源和接口来扩展API。
    - 独立性：每个请求都是独立的，服务器不保存请求的状态信息。
缺点：
    - 难以实现复杂的业务逻辑：RESTful API更适合简单的业务逻辑。
    - 安全性：由于RESTful API是基于HTTP协议的，所以需要额外的安全机制来保护API。
```

## HTTP问题
**定义**
```
HTTP（HyperText Transfer Protocol，超文本传输协议）是一种用于传输超文本（例如 HTML）数据的应用层协议。它是基于客户端-服务器架构工作的，客户端发送 HTTP 请求，服务器返回 HTTP 响应。
```
**HTTP协议的特点**
```
1. 无连接：HTTP 协议是无连接的，即每次请求都需要建立新的连接，请求结束后立即关闭连接，每个连接只处理一个请求。
2. 无状态：服务器不会保存任何关于客户端的信息，每个请求都是独立的，服务器不知道这是同一个客户端的请求还是不同客户端的请求。
3. 简单快速：简单易懂，通信速度快，适用于传输文本、图片等数据。
4. 灵活：HTTP 允许传输任意类型的数据，不限于 HTML 文档，也可以传输图片、视频、音频等多媒体数据。
```
**HTTP请求**
```
HTTP 协议常见的请求方法有 GET、POST、PUT、DELETE 等，用来指定请求的类型。常见的状态码有 200（OK）、404（Not Found）、500（Internal Server Error）等，用来表示服务器对请求的处理结果。

HTTP 请求由以下部分组成：
	1. 请求行（Request Line）：包括请求方法、请求的 URL 和协议版本。
	2. 请求头部（Request Headers）：包括一些关于客户端和请求的信息，如 User-Agent、Host、Accept 等。
	3. 空行（Blank Line）：用来分隔请求头部和请求体。
	4. 请求体（Request Body）：包含请求的数据，如 POST 请求中的表单数据。
```
**HTTP响应**
```
HTTP 响应由以下部分组成：
	1. 状态行（Status Line）：包括协议版本、状态码和状态消息。
	2. 响应头部（Response Headers）：包括一些关于响应的信息，如 Server、Content-Type、Content-Length 等。
	3. 空行（Blank Line）：用来分隔响应头部和响应体。
	4. 响应体（Response Body）：包含响应的数据，如 HTML 文档、图片等。
```

**GET和POST区别**
```
GET 适合用于获取数据，而 POST 适合用于提交数据
- GET 请求将数据包含在 URL 中，以查询字符串的形式发送，例如 `http://example.com/page?name=value`。对 URL 长度有限制，不适合发送大量数据。在 URL 中可见，不适合发送敏感信息。可以被浏览器缓存，可以通过浏览器历史记录查看。幂等的，即多次发送同样的 GET 请求会产生相同的结果，不会对服务器产生影响。

- POST 请求将数据包含在请求体中发送，适用于发送大量数据或敏感信息。没有长度限制，适用于发送大量数据。相对安全.不会被浏览器缓存，数据不会保存在历史记录中。不是幂等的，多次发送同样的 POST 请求可能会产生不同的结果，会对服务器产生影响。
```

**不同语言的前端，后端，客户端，将这三个协作起来有什么方式**
```
1. 前后端分离（Frontend-Backend Separation）:前端负责页面展示和用户交互，后端负责业务逻辑和数据处理。前端通过 AJAX、Fetch API 或 WebSocket 等与后端通信，获取数据和执行操作。后端提供 RESTful API 或 GraphQL 接口，用于前端向后端发送请求和接收响应。
2. 模板引擎（Template Engine）：后端使用模板引擎生成动态页面，前端负责展示。前端通过 AJAX 或表单提交数据到后端，后端处理数据并返回渲染后的页面。
3. 服务器端渲染（Server-Side Rendering，SSR）：后端负责渲染页面的 HTML，包含动态数据。客户端（浏览器）接收到渲染好的 HTML 页面后，负责展示和交互。
4. 客户端渲染（Client-Side Rendering，CSR）：前端框架（如 Angular、React、Vue.js）在客户端渲染页面，包括数据获取和展示。后端只提供数据接口，不负责页面渲染。
5. WebSocket 实时通信：实现客户端和服务器之间的实时双向通信。通过 WebSocket，前端可以向后端发送消息并接收实时更新的数据，实现实时性要求较高的功能。
```
URL输入到显示的过程
```
URL（统一资源定位符）输入到显示的过程涉及多个步骤，主要包括域名解析、建立连接、发送请求和响应页面渲染等。以下是详细的过程：

1. 域名解析：
    - 用户在浏览器中输入 URL（例如 https://www.example.com）。
    - 浏览器解析 URL，提取出主机名（www.example.com）。
    - 浏览器向本地 DNS 服务器发送域名解析请求，获取主机名对应的 IP 地址。
2. 建立连接：
    - 浏览器通过 IP 地址和端口号（例如 HTTPS 默认端口号为 443）向服务器发起 TCP 连接请求。
    - 服务器接受连接请求，建立 TCP 连接。
3. 发送请求：
    - 浏览器向服务器发送 HTTP 请求，包括请求方法（GET、POST 等）、路径（/）、请求头（User-Agent、Accept 等）和请求体（POST 请求时包含的数据）。
4. 处理请求：
    - 服务器接收到请求后，根据路径和请求方法执行相应的处理逻辑，可能涉及查询数据库、调用后端服务等。
5. 生成响应：
    - 服务器处理完请求后，生成 HTTP 响应，包括状态码（200、404 等）、响应头（Content-Type、Content-Length 等）和响应体（HTML、JSON 等）。
6. 传输响应：
    - 服务器将响应通过 TCP 连接传输给浏览器。
7. 渲染页面：
    - 浏览器接收到响应后，根据响应头中的 Content-Type 等信息判断响应内容的类型。
    - 如果是 HTML 页面，浏览器解析 HTML、CSS 和 JavaScript，构建文档对象模型（DOM）和渲染树（Render Tree），最终将页面呈现给用户。
8. 页面显示：
    - 浏览器根据 DOM 和渲染树计算每个元素的位置和样式，并将页面内容显示在浏览器窗口中。
9. 完整页面加载：
    - 页面加载完成后，浏览器可能还会加载页面中的图片、样式表、脚本等资源，直到页面完全加载完成。
```
